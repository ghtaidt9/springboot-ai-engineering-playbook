# ADR-003: Hexagonal Architecture with Internal Layering

**Status:** Accepted
**Date:** 2026-06-25
**Deciders:** Engineering team

---

## Context

Two architecture models existed in the codebase simultaneously:
- `AGENTS.md`, `architecture.mdc`, `system-overview.md` → **Hexagonal**
- `backend-architecture.md` → **Traditional Layered** (Controller → Service → Repository)

This caused the agent to generate inconsistent package structures.

---

## Decision

**Hexagonal Architecture is the outer boundary. Standard layering applies *inside* each adapter.**

In practice:
- The domain and port interfaces are hexagonal (no framework dependencies)
- Inside `adapter/in/web/`, the controller follows standard Spring MVC layering
- Inside `adapter/out/persistence/`, JPA conventions apply

This is not "hexagonal vs layered" — it is **hexagonal containing layered adapters**.

---

## Rationale

Pure hexagonal is more correct for the domain; pure layered is easier for adapter internals. The hybrid gives both benefits without sacrificing consistency.

---

## Consequences

- Package structure is always `domain/`, `application/`, `adapter/in/`, `adapter/out/`
- `@Service` annotation belongs on **application services**, not domain services
- `@Repository` annotation belongs in `adapter/out/persistence/` only
