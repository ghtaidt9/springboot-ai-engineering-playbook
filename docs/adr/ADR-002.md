# ADR-002: Standardize on Spring Data JPA

**Status:** Accepted
**Date:** 2026-06-25
**Deciders:** Engineering team

---

## Context

The codebase had conflicting persistence approaches:
- `java.mdc` and `backend-architecture.md` referenced **MyBatis**
- `persistence.md` defined patterns for **Spring Data JPA / Hibernate**

Both cannot coexist as the default — agent-assisted coding requires a single pattern to follow.

---

## Decision

**Use Spring Data JPA as the sole ORM for all services in this playbook.**

MyBatis references are removed from all docs and rules.

---

## Rationale

| Criterion | Spring Data JPA | MyBatis |
|-----------|----------------|---------|
| Spring Boot integration | Native, zero config | Requires extra setup |
| Domain-driven approach | Entity ↔ domain mapping natural | SQL-centric, leaks schema into domain |
| Boilerplate for CRUD | Minimal (repository interface) | Manual SQL for every operation |
| Type safety | Yes (JPQL / Criteria) | String SQL |
| Playbook audience | Spring Boot developers | Fits Java EE / legacy context more |

---

## Consequences

**Positive:**
- Single pattern → agent generates consistent code
- Flyway + JPA = standard Spring Boot stack with broad community support

**Negative:**
- Complex reporting queries may need raw SQL workaround via `@Query` projections
- Teams familiar with MyBatis need to learn JPA

**Mitigation:** If a specific service genuinely needs MyBatis for read-heavy analytics, create `ADR-002b` for that service scope only.
